# RAG（检索增强生成）知识库设计方案

**适用产品**：校园AI秘书  
**文档版本**：V1.0  
**设计日期**：2024年4月

---

## 一、概述

本方案设计用于支持校园AI秘书的智能信息解析与日程生成功能。通过构建结构化知识库，结合检索增强生成技术，实现对课程通知、社团活动等非结构化信息的精准解析，确保AI回答的准确性和可追溯性。

---

## 二、知识库数据来源设计

### 2.1 数据源分类

| 数据类别 | 来源 | 更新频率 | 数据格式 | 关键字段 |
|---------|------|---------|---------|---------|
| 个人课表 | 教务系统/用户上传 | 学期初 | JSON/图片 | 课程名、时间、地点、教师 |
| 教务通知 | 教务系统/官网 | 实时 | HTML/文本 | 标题、时间、地点、类型 |
| 社团公告 | QQ群/公众号/小程序 | 实时 | 文本/图片 | 活动名称、时间、地点、主办方 |
| 校园日历 | 校历/活动日历 | 定期 | JSON/ICS | 事件名称、时间、类型 |
| 考试安排 | 教务系统 | 考前 | HTML/文本 | 科目、时间、地点、座位号 |
| 用户日程 | 用户手动创建 | 实时 | JSON | 事件名、时间、地点、优先级 |
| 校园设施 | 后勤系统 | 定期 | JSON | 设施名、开放时间、位置 |

### 2.2 数据采集策略

**自动化采集**：
- 教务系统：通过API接口定时同步（每日0点）
- QQ群消息：监听指定群聊，自动抓取公告类消息
- 公众号：订阅校园官方公众号，自动提取推文内容

**用户主动上传**：
- 课表截图：用户发送图片，OCR识别后结构化存储
- 转发消息：用户转发群通知/邮件，自动解析存储

**手动录入**：
- 用户手动添加的个人日程
- 管理员维护的校园设施信息

## 三、文档切片策略

### 3.1 分块原则

| 文档类型 | 分块策略 | 块大小 | 重叠率 |
|---------|---------|-------|-------|
| 短通知（<500字） | 全文作为一个块 | 完整文档 | 0% |
| 长通知（500-2000字） | 按段落分块 | 300-500字 | 10-15% |
| 课表图片 | 按课程条目分块 | 单节课信息 | 0% |
| 考试安排 | 按考试条目分块 | 单场考试信息 | 0% |
| 校园日历 | 按事件条目分块 | 单个事件 | 0% |

### 3.2 元数据设计

每个切片块包含以下元数据：

| 元字段 | 类型 | 说明 | 示例 |
|-------|------|------|------|
| chunk_id | UUID | 唯一标识 | `a1b2c3d4-...` |
| source_type | 枚举 | 数据来源类型 | `教务通知` |
| source_url | 字符串 | 原始来源链接 | `http://jwxt.xxx.edu.cn/...` |
| publish_time | 时间戳 | 发布时间 | `2024-01-15T09:00:00` |
| entity_tags | 数组 | 实体标签 | `["高等数学", "教学楼A302"]` |
| priority | 枚举 | 优先级 | `high/medium/low` |
| valid_period | 对象 | 有效时间段 | `{start, end}` |

### 3.3 存储结构

```
knowledge_base/
├── raw_data/           # 原始数据存储
│   ├── notices/        # 通知文档（HTML/Markdown）
│   ├── schedules/      # 课表数据（JSON）
│   └── images/         # 图片文件（课表截图等）
├── vector_db/          # 向量数据库
│   ├── embeddings/     # 文本嵌入向量
│   └── metadata/       # 元数据索引
└── search_index/       # 关键词索引
    ├── inverted_index/ # 倒排索引
    └── entity_index/   # 实体索引
```

## 四、检索策略

### 4.1 混合检索架构

```
用户查询 → 关键词检索 → 语义检索 → 结果融合 → 排序输出
              ↓              ↓
         倒排索引查询    向量相似度匹配
```

### 4.2 关键词检索

**索引构建**：
- 对每个切片块建立倒排索引
- 提取实体词（课程名、地点、人名）作为关键词
- 支持同义词扩展（如"教学楼"="教学大楼"）

**查询处理**：
- 分词处理：使用Jieba分词
- 停用词过滤：去除"的"、"是"等无意义词
- 关键词权重：根据词频和重要性分配权重

### 4.3 语义检索

**嵌入模型选择**：
- 使用 `text-embedding-3-small` 模型（OpenAI）
- 向量维度：1536维
- 支持中英文混合文本

**相似度计算**：
- 采用余弦相似度度量
- 阈值设置：相似度≥0.7视为相关

**检索范围**：
- 优先检索时间相关文档（近30天）
- 根据用户上下文过滤（如当前学期课表）

### 4.4 结果融合策略

| 检索方式 | 权重 | 作用 |
|---------|------|------|
| 关键词匹配 | 40% | 确保字面匹配准确性 |
| 语义相似度 | 40% | 捕捉上下文相关性 |
| 时间相关性 | 10% | 优先近期信息 |
| 来源可信度 | 10% | 优先官方渠道 |

### 4.5 检索流程

```
1. 接收用户输入（文字/图片/转发消息）
2. 解析输入内容，提取查询关键词
3. 执行关键词检索，获取候选文档
4. 对候选文档进行语义匹配
5. 融合两种检索结果，按综合得分排序
6. 返回Top-K相关文档（K=3~5）
```

## 五、提示词模板设计

### 5.1 系统Prompt完整示例

```
你是一位专业的校园AI秘书，擅长处理大学生的日程管理需求。

【你的角色】
- 身份：可靠、高效的校园生活助手
- 语气：友好、简洁、专业
- 目标：帮助用户管理日程、不错过重要事项

【核心能力】
1. 解析课表截图、群通知、邮件等信息
2. 自动提取时间、地点、事件等实体
3. 生成结构化日程并设置提醒
4. 回答校园相关问题

【知识库使用规则】
- 回答必须基于提供的知识库内容
- 如果知识库中没有相关信息，明确告知用户
- 优先使用最新的信息（检查publish_time）
- 对于不确定的信息，说明信息来源和时间

【实体识别要求】
- 时间：解析为标准格式 YYYY-MM-DD HH:MM
- 地点：提取具体位置（如"教学楼A302"）
- 事件名称：简洁描述（如"高等数学期末考试"）
- 优先级：根据事件类型自动判断（考试>作业>活动）

【输出格式要求】
1. 日程解析结果：
   - 事件名称：[事件描述]
   - 时间：[开始时间] - [结束时间]
   - 地点：[具体位置]
   - 来源：[信息来源]
   - 优先级：[高/中/低]

2. 问答回答：
   - 直接回答问题，引用知识库内容
   - 如需生成日程，提供选项供用户确认

【边界规则】
- 不回答与校园生活无关的问题
- 不提供学术作弊相关服务
- 不生成未经证实的信息
- 对不确定的信息明确说明

【用户上下文】
当前时间：{current_time}
用户学期：{current_semester}
用户课表：{user_schedule}
用户已创建日程：{user_events}

请根据以上规则，基于提供的知识库内容，处理用户请求。
```

## 六、幻觉控制策略

### 6.1 检索约束

**强制检索机制**：
- 所有回答必须基于检索到的知识库内容
- 如果检索结果为空或相似度低于阈值，拒绝回答
- 明确告知用户"暂无相关信息，请查阅官方渠道"

**检索范围限制**：
- 根据用户上下文过滤（如当前学期、所在校区）
- 时间范围限制（优先近期信息）
- 来源可信度排序（教务系统>官方公众号>社团群）

### 6.2 内容验证

**事实核查机制**：
- 对涉及考试时间、地点等关键信息，强制交叉验证
- 对比多个来源的一致性
- 标记冲突信息并提示用户确认

**时间有效性检查**：
- 检查文档的valid_period字段
- 对过期信息标注"已过期"
- 优先使用最新发布的信息

### 6.3 输出约束

**格式强制**：
- 日程输出必须包含"来源"字段
- 问答回答必须引用知识库内容
- 不确定的信息必须明确标注

**置信度标记**：
- 根据检索结果质量添加置信度标签
- 高置信度（≥0.85）：直接回答
- 中置信度（0.7-0.85）：注明"信息来源：xxx"
- 低置信度（<0.7）：拒绝回答，引导官方渠道

### 6.4 反馈机制

**用户反馈通道**：
- 提供"信息有误"反馈按钮
- 用户可标记错误信息并说明原因

**错误纠正流程**：
```
用户反馈错误 → 人工审核 → 更新知识库 → 通知用户更正
```

**模型迭代**：
- 定期分析错误案例
- 优化检索策略和提示词
- 更新实体识别规则

## 七、性能优化策略

### 7.1 索引优化
- 使用分层索引结构（年度→学期→月份）
- 建立热点数据缓存（近期通知、用户课表）
- 增量更新索引，避免全量重建

### 7.2 检索优化
- 设置合理的检索范围（时间、来源类型）
- 使用近似最近邻（ANN）算法加速向量检索
- 缓存高频查询结果

### 7.3 系统架构
- 采用微服务架构，检索与生成分离
- 使用Redis缓存热点数据
- 异步处理非实时任务（如索引更新）

## 八、安全与合规

### 8.1 数据安全
- 用户课表、日程采用加密存储
- 传输数据使用HTTPS/TLS加密
- 访问控制：基于角色的权限管理

### 8.2 隐私保护
- 不存储用户聊天记录原文
- 仅存储必要的结构化信息
- 用户可随时删除个人数据

### 8.3 合规要求
- 符合《个人信息保护法》
- 获得用户明确授权后使用数据
- 数据留存期限不超过必要时长
